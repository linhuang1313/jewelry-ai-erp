4. 修改前未完整读取和理解相关代码段（约250行）
5. 修改后未立即运行 linter 检查

**修复方法：**
```javascript
// 删除了多余的代码块
// 第385-390行的多余代码：
// }
//   
//   setMessages(prev => [...prev, { 
//     type: 'system', 
//     content: systemMessage 
//   }])

**经验教训：**
1. **修改前完整理解代码结构**：修改前应该完整读取相关代码段，理解整体逻辑结构

## 2. 数据库相关问题

### 错误 #2: 数据库迁移误解

**错误描述：**
- 用户询问："为什么需要数据库迁移呢？我记得我已经有一个入库数据库了？"
- 误解：认为添加新表需要数据库迁移工具

**错误原因：**
1. 对 SQLAlchemy 的 `create_all()` 行为理解不准确
2. 误以为每次添加新表都需要使用 Alembic 等迁移工具
3. 没有明确说明 `Base.metadata.create_all(bind=engine)` 会自动创建缺失的表，不会影响现有表

**修复方法：**
- 澄清：`create_all()` 只会创建不存在的表，不会删除或修改现有表
- 对于新表，直接调用 `init_db()` 即可，不需要迁移
- 只有在需要修改现有表结构时，才需要使用 Alembic

**经验教训：**
1. **准确理解工具行为**：在使用工具前，应该准确理解其行为机制
2. **向用户解释清楚**：当用户有疑问时，应该详细解释技术细节
3. **区分场景**：新表创建 vs 表结构修改，使用不同的策略

**如何避免：**
- ✅ 在使用工具前查阅文档，准确理解行为
- ✅ 向用户解释技术细节，避免误解
- ✅ 区分不同场景，使用合适的工具

## 3. 业务逻辑错误

### 错误 #3: 库存检查顺序错误

**错误描述：**
- 文件：`backend/app/main.py` - `create_sales_order` 函数
- 问题：库存检查在客户创建之后执行
- 影响：如果库存不足，会导致创建了客户但销售单创建失败，产生孤儿记录

**错误原因：**
1. 没有理解业务逻辑的依赖关系
2. 先创建客户再检查库存，违反了"先验证后操作"的原则
3. 没有考虑事务的原子性

**修复方法：**
```python
# 修复前：先创建客户，再检查库存
customer = create_customer(...)  # 先创建
check_inventory(...)  # 后检查

# 修复后：先检查库存，再创建客户
check_inventory(...)  # 先检查
if inventory_errors:
    return error  # 如果库存不足，直接返回，不创建客户
customer = create_customer(...)  # 库存充足后再创建
```

**经验教训：**
1. **理解业务逻辑依赖**：先验证所有前置条件，再执行操作
2. **考虑事务原子性**：确保操作的原子性，要么全部成功，要么全部失败
3. **先验证后操作**：始终遵循"先验证后操作"的原则

**如何避免：**
- ✅ 在实现业务逻辑前，先理解依赖关系
- ✅ 先验证所有前置条件，再执行操作
- ✅ 考虑事务的原子性，避免部分成功的情况

### 错误 #4: 库存预留逻辑缺失

**错误描述：**
- 文件：`backend/app/main.py` - `create_sales_order` 函数
- 问题：库存检查时没有考虑已预留的库存（"待结算"状态的销售单）
- 影响：可能导致超卖，多个销售单同时占用同一批库存

**错误原因：**
1. 没有理解业务场景：销售单创建时状态为"待结算"，此时库存应该被预留
2. 只检查了总库存，没有检查可用库存（总库存 - 已预留库存）
3. 没有考虑并发场景

**修复方法：**
```python
# 修复前：只检查总库存
available_weight = inventory.total_weight

# 修复后：检查可用库存（总库存 - 已预留库存）
reserved_weight = db.query(func.sum(SalesDetail.weight)).join(
    SalesOrder
).filter(
    SalesOrder.status == "待结算",
    SalesDetail.product_name == item.product_name
).scalar() or 0

available_weight = inventory.total_weight - reserved_weight
```

**经验教训：**
1. **理解业务场景**：销售单的"待结算"状态意味着库存被预留
2. **考虑并发场景**：多个操作可能同时进行，需要预留机制
3. **完整的数据模型**：需要考虑所有影响库存的因素

**如何避免：**
- ✅ 深入理解业务场景，考虑所有状态
- ✅ 考虑并发场景，使用预留机制
- ✅ 完整的数据模型，考虑所有影响因素

if item.weight <= 0:

        "product_name": item.product_name,

if item.labor_cost < 0:

        "product_name": item.product_name,
        "error": "工费不能为负数",
        ...

**经验教训：**

2. **防御性编程**：假设输入可能无效，进行验证
3. **清晰的错误信息**：提供详细的错误信息，帮助用户理解问题

**如何避免：**
- ✅ 始终验证所有输入数据

- ✅ 提供清晰的错误信息

### 错误 #6: 日期解析错误处理缺失

**错误描述：**
- 文件：`backend/app/main.py` - `handle_create_sales_order` 函数
- 问题：`datetime.fromisoformat` 可能因为无效日期格式而抛出异常
- 影响：如果用户提供无效日期格式，会导致整个请求失败

**错误原因：**
1. 没有处理日期解析可能的异常
2. 假设日期格式总是有效的
3. 没有提供默认值或回退机制

**修复方法：**
```python
# 修复前：直接解析，可能抛出异常
order_date = datetime.fromisoformat(ai_response.order_date)

# 修复后：添加异常处理
try:
    order_date = datetime.fromisoformat(ai_response.order_date)
except (ValueError, AttributeError):
    logger.warning(f"无效的日期格式: {ai_response.order_date}，使用当前时间")
    order_date = datetime.now()
```

**经验教训：**
1. **处理可能的异常**：对于可能失败的操作，总是添加异常处理
2. **提供默认值**：当解析失败时，提供合理的默认值
3. **记录警告**：记录警告信息，便于调试

**如何避免：**
- ✅ 对可能失败的操作添加异常处理
- ✅ 提供合理的默认值或回退机制
- ✅ 记录警告信息，便于调试

## 5. AI集成问题

### 错误 #7: AI上下文理解不足

**错误描述：**
- 文件：`backend/app/ai_parser.py`
- 问题：AI无法回答后续问题，如"哪七种"（在回答"有7种商品"之后）
- 影响：用户体验差，需要重复提问

**错误原因：**
1. AI解析器的提示词没有包含上下文追问识别的规则
2. 没有考虑对话的上下文连续性
3. 没有处理简短的追问（如"哪七种"、"有哪些"等）

**修复方法：**
```python
# 在提示词中添加上下文追问识别规则
prompt = f"""
5. **上下文追问识别**（非常重要）：
   - 当用户使用"哪X种"、"有哪些"、"列出所有"、"具体是哪些"、"哪几个"、"都有什么"等追问时，需要根据上下文判断：
     * 如果涉及"商品"、"库存"、"种类"、"种商品" → action: "查询库存"
     * 如果涉及"供应商"、"哪几个供应商" → action: "查询供应商"
     * 如果涉及"客户"、"哪几个客户" → action: "查询客户"
   - **特别重要**：当用户单独说"哪七种"、"哪几种"、"哪几个"、"有哪些"、"具体是哪些"等简短追问时：
     * 如果包含"种"字（如"哪七种"、"哪几种"、"有几种"），通常是指商品种类，应该识别为 action: "查询库存"
"""
```

**经验教训：**
1. **考虑对话上下文**：AI系统需要理解对话的连续性
2. **处理简短追问**：用户可能使用简短的追问，需要识别意图
3. **完善的提示词**：提示词应该包含所有可能的场景和规则

**如何避免：**
- ✅ 考虑对话的上下文连续性
- ✅ 处理各种可能的追问方式
- ✅ 完善提示词，包含所有场景

**错误原因：**

2. 误以为每次添加新表都需要使用 Alembic 等迁移工具

- 对于新表，直接调用 `init_db()` 即可，不需要迁移
- 只有在需要修改现有表结构时，才需要使用 Alembic

**经验教训：**
1. **准确理解工具行为**：在使用工具前，应该准确理解其行为机制

available_weight = inventory.total_weight

reserved_weight = db.query(func.sum(SalesDetail.weight)).join(
